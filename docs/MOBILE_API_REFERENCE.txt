================================================================================
MOBILE EMPLOYEE API - PAYLOAD & RESPONSE REFERENCE
================================================================================

Base URL:  /api/mobile
Audit:     All successful state-changing actions emit an AuditLog entry
           via createAuditLog (fire-and-forget, never blocks the response).

Common envelope (success):
{
  "success": true,
  "data": { ... }
}

Common envelope (error):
{
  "success": false,
  "error": {
    "code": "VALIDATION",
    "message": "First name is required"
  }
}

Error codes used across the mobile API:
  UNAUTHORIZED          401  Missing / invalid / expired Supabase token
  ACCOUNT_DISABLED      403  Account.status != ACTIVE
  ROLE_NOT_ALLOWED      403  Account.role != EMPLOYEE
  EMPLOYEE_INACTIVE     403  Employee.status != ACTIVE (or soft-deleted)
  COMPANY_INACTIVE      403  Company is CANCELLED / PAUSED / SUSPENDED
  FORBIDDEN             403  Cross-role access denied
  VALIDATION            400  Bad input
  NOT_FOUND             404  Resource does not exist
  INTERNAL              500  Unhandled exception (never exposes Prisma)

================================================================================
MOBILE AUTHENTICATION ARCHITECTURE
================================================================================

1. Single authentication provider
---------------------------------
Supabase is the ONLY authentication provider for the mobile app.
The backend does NOT generate, store, or sign custom JWTs. There is
no application token system. Supabase access tokens are the only
credentials the mobile app needs.

2. Login flow
-------------
a. The mobile app calls the Supabase Auth SDK (signInWithPassword,
   signInWithIdToken, OAuth, etc.). Supabase returns:
     - access_token       (short-lived, default 1h)
     - refresh_token      (long-lived, default 30d)
     - user.id            (UUID, immutable, == Account.authUserId)
     - user.email
b. The mobile app stores both tokens securely on the device
   (Keychain on iOS, EncryptedSharedPreferences on Android — never
   in plain SharedPreferences, UserDefaults, AsyncStorage, or logs).
c. The mobile app calls POST /api/mobile/auth/sync with
   Authorization: Bearer <access_token> to bridge the Supabase user
   with the application Account / Employee / Company records.
d. The sync endpoint returns the employee profile. The app is now
   logged in. No application JWT is ever issued or returned.

3. Authorization header
-----------------------
Every protected mobile endpoint requires:

  Authorization: Bearer <access_token>

The backend validates the token via `supabase.auth.getUser(token)`,
then loads the application Account by `authUserId = user.id`. There
is NO support for cookie-based auth on /api/mobile/* — Bearer is
the only transport. (The web app uses cookies; the mobile app does
not.)

4. Reusable helper — getAuthenticatedMobileEmployee(request)
-----------------------------------------------------------
File: src/lib/mobile-auth.ts

Every protected mobile route uses this helper. It:
  1. Reads the Authorization: Bearer header
  2. Validates the token via supabase.auth.getUser(token)
  3. Finds the application Account by authUserId
  4. Verifies Account.status === 'ACTIVE'
  5. Verifies Account.role   === 'EMPLOYEE'
  6. Loads the Employee via the Account.employee relation
  7. Verifies Employee.status === 'ACTIVE' AND deletedAt == null
  8. Loads the Company
  9. Verifies the Company is ACTIVE (rejects CANCELLED / PAUSED /
     SUSPENDED / soft-deleted)
 10. Returns { ok: true, user, account, employee, company } on success
     or { ok: false, response } with a ready-to-emit NextResponse
     on failure

Usage in a route:
  const auth = await getAuthenticatedMobileEmployee(request)
  if (!auth.ok) return auth.response
  const { user, account, employee, company } = auth

5. Token lifecycle and refresh
------------------------------
- The Supabase access_token has a 1-hour expiry by default.
- When a mobile endpoint returns 401 UNAUTHORIZED, the mobile app
  MUST call supabase.auth.refreshSession() to obtain a new
  access_token + refresh_token, persist them, and retry the
  original request once. If refresh fails, the user is signed out
  and routed to the login screen.
- The mobile app MUST treat both tokens as opaque strings. They
  must never be decoded client-side.
- The mobile app MUST NEVER log the access_token or refresh_token,
  even in debug builds.
- POST /api/mobile/auth/logout calls supabase.auth.signOut()
  server-side, which invalidates the refresh token. The mobile app
  should clear both tokens from secure storage on success.

6. Authentication error responses
---------------------------------
The helper emits one of the following error responses. All routes
inherit the same envelope and codes — do not introduce new ones.

  401 UNAUTHORIZED
    Missing Bearer token
      { "success": false,
        "error": { "code": "UNAUTHORIZED",
                   "message": "Missing Bearer token" } }

  401 UNAUTHORIZED
    Invalid or expired token
      { "success": false,
        "error": { "code": "UNAUTHORIZED",
                   "message": "Invalid or expired token" } }

  403 ACCOUNT_NOT_MAPPED
    No application Account row for the Supabase user
      { "success": false,
        "error": { "code": "ACCOUNT_NOT_MAPPED",
                   "message": "You are not mapped to any account. Please contact your administrator." } }

  403 ACCOUNT_DISABLED
    Account.status != ACTIVE
      { "success": false,
        "error": { "code": "ACCOUNT_DISABLED",
                   "message": "Account is inactive or suspended." } }

  403 ROLE_NOT_ALLOWED
    Account.role != EMPLOYEE
      { "success": false,
        "error": { "code": "ROLE_NOT_ALLOWED",
                   "message": "This endpoint is only available for employee accounts." } }

  404 EMPLOYEE_NOT_FOUND
    No Employee row linked to the Account
      { "success": false,
        "error": { "code": "EMPLOYEE_NOT_FOUND",
                   "message": "No employee profile is linked to this account." } }

  403 EMPLOYEE_INACTIVE
    Employee.status != ACTIVE (or deletedAt != null)
      { "success": false,
        "error": { "code": "EMPLOYEE_INACTIVE",
                   "message": "Your account is not active. Please contact your administrator.",
                   "employeeStatus": "SUSPENDED" } }

  403 COMPANY_INACTIVE
    Company is CANCELLED / PAUSED / SUSPENDED / soft-deleted
      { "success": false,
        "error": { "code": "COMPANY_INACTIVE",
                   "message": "Your company's access is currently inactive.",
                   "companyStatus": "PAUSED" } }

  500 INTERNAL
    Unexpected exception (never exposes Prisma)
      { "success": false,
        "error": { "code": "INTERNAL",
                   "message": "Internal server error" } }

7. Protected mobile endpoints
-----------------------------
All routes below require `Authorization: Bearer <access_token>` and
return 401 UNAUTHORIZED on a missing / invalid token.

  Authentication / session
    POST   /api/mobile/auth/sync
    POST   /api/mobile/auth/logout

  Home feed
    GET    /api/mobile/employee/home

  Offers
    GET    /api/mobile/offers
    GET    /api/mobile/offers/[id]
    POST   /api/mobile/offers/[id]/click
    POST   /api/mobile/offers/[id]/save
    DELETE /api/mobile/offers/[id]/save
    POST   /api/mobile/offers/[id]/redeem

  Brands
    GET    /api/mobile/brands
    GET    /api/mobile/brands/[id]

  Categories
    GET    /api/mobile/categories

  Profile
    GET    /api/mobile/profile
    PATCH  /api/mobile/profile

  Notifications
    GET    /api/mobile/notifications
    PATCH  /api/mobile/notifications/read

The only mobile route that is NOT behind the Bearer check is
POST /api/mobile/auth/sync — but even there, the Bearer token is
REQUIRED (the mobile app must sign in at Supabase first and then
pass that token to /sync).

1. POST /api/mobile/auth/sync
================================================================================

Purpose:  Bridge the Supabase Auth user with the application employee
          record. The mobile app calls this immediately after Supabase
          signIn / on app resume.

Auth:     Bearer token (Supabase access_token). userType must be 'employee'.

Request:
  POST /api/mobile/auth/sync
  Content-Type: application/json

  Body (all fields optional):
  

Response 200:
  {
    "success": true,
    "data": {
      "employeeId":  "7c1a3e62-9a40-4f4d-bd11-3f8a0c5a2e91",
      "firstName":   "Alice",
      "lastName":    "Williams",
      "email":       "alice@techcorp.com",
      "avatarUrl":   "https://cdn.example.com/avatars/alice.jpg",
      "company": {
        "id":     "0c4b9c12-3e5b-4f2a-8b6e-1f2d3c4b5a67",
        "name":   "Tech Corp",
        "status": "ACTIVE"
      },
      "role": "EMPLOYEE"
    }
  }

Response 401:
  { "success": false, "error": { "code": "UNAUTHORIZED", "message": "Unauthorized" } }

Response 403 (employee inactive):
  {
    "success": false,
    "error": {
      "code": "EMPLOYEE_INACTIVE",
      "message": "Your account is not active. Please contact your administrator.",
      "employeeStatus": "SUSPENDED"
    }
  }

Response 403 (company cancelled):
  { "success": false, "error": { "code": "COMPANY_INACTIVE", "message": "Company is not active" } }

Response 404 (employee record missing):
  { "success": false, "error": { "code": "EMPLOYEE_NOT_FOUND", "message": "Employee not found" } }

Audit:   MOBILE_AUTH_SYNC
Side effects:
  - Account.lastLoginAt = now()
  - Employee.lastLoginAt = now()

2. POST /api/mobile/auth/logout
================================================================================

Purpose:  Properly end the mobile session. Calls supabase.auth.signOut()
          server-side (invalidation of the refresh token) and clears
          the session cookies.

Auth:     Bearer token (any authenticated user; the logout endpoint accepts any valid Supabase user).

Request:
  POST /api/mobile/auth/logout
  (no body)

Response 200:
  { "success": true }

Side effects:
  - supabase.auth.signOut() is called
  - sb-access-token cookie deleted
  - sb-refresh-token cookie deleted

Audit:   MOBILE_LOGOUT

  Content-Type: application/json

  {

    "branchId":    "b1a2...uuid",    // optional, must belong to offer's merchant

  {
    "success": true,
    "data": {
      "id":             "r1a2...uuid",

      "method":         "IN_STORE",
      "status":         "PENDING",
      "message":        "Redemption submitted. Show this code to the merchant."
    }
  }

Response 400 (validation):
  { "success": false, "error": { "code": "VALIDATION", "message": "method must be one of: IN_STORE, ONLINE, QR_CODE, MANUAL_CODE" } }

  { "success": false, "error": { "code": "VALIDATION", "message": "You have already redeemed this offer" } }

Response 400 (bad branch):
  { "success": false, "error": { "code": "VALIDATION", "message": "Invalid branchId for this offer" } }

Side effects:
  - Redemption row created (isVerified=false)
  - MerchantOffer.currentRedemptions += 1
